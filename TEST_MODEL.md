# Тестовая модель приложения pdflashcards

## 1. Критические пользовательские сценарии
1. Регистрация пользователя.
2. Вход в систему и получение access/refresh token.
3. Доступ к защищённым страницам и endpoint'ам только после авторизации.
4. Загрузка PDF-файла и сохранение записи в истории.
5. Получение списка загрузок, фильтрация, поиск, сортировка, пагинация.
6. Генерация карточек по PDF.
7. Просмотр карточек только владельцем файла или администратором.
8. Работа со связанными книгами через внешний API.
9. Выход из системы и запрет повторного использования refresh token.

## 2. Ключевые бизнес-правила и ограничения
1. Первый зарегистрированный пользователь получает роль admin.
2. Остальные пользователи получают роль user.
3. Пользователь не должен видеть и изменять чужие файлы.
4. Admin может выполнять административные операции.
5. Разрешён только PDF.
6. Пустой файл запрещён.
7. Максимальный размер PDF — 10 МБ.
8. Название файла в истории не может быть пустым.
9. У одного пользователя не должно быть двух записей с одинаковым title.
10. После logout refresh token должен считаться недействительным.

## 3. Области повышенного риска
1. Аутентификация и refresh token flow.
2. Роли и права доступа.
3. Загрузка, удаление и очистка файлов.
4. Интеграция с объектным хранилищем.
5. Интеграция с внешним API книг.
6. Генерация карточек через внешнюю модель.
7. Обработка 401/403/404/409 и ошибок внешних сервисов.
